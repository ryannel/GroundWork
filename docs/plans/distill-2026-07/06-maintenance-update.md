## 1. Intention

The section is GroundWork's answer to "how does an already-installed project stay true — to its own code, and to the framework it was installed from?" Three currents: the framework moves (update lane: CLI mechanics + `groundwork-update` judgment), the project's code moves (doc-sync), and the product grows a new surface (surface-activation). The deliberate architecture is a **mechanics/judgment split**: everything deterministic lives in `bin/groundwork.js` + `migrations/`; everything needing eyes lives in the `groundwork-update` skill (Phase A brief, Phase B Family Index).

Per-file intention map:

- **`groundwork-update/instructions.md`** — make the agent behave as a *thin driver*: work the CLI-staged brief item-by-item, then detect structural drift per family and advance it, farming every heavy read/transform to a disposable worker so the driver keeps enough context to pace the user, run the review gate, and commit one unit at a time. End state: "the project looks as if it was set up at the current version all along."
- **`groundwork-update/briefs/reconcile-worker.md`** — a self-contained subagent brief: given one unit capsule, read canonical + instances in full, advance honestly by the matching recipe, never commit, and return a tight structured report (FILES / REVIEW-NEEDED / COLLISIONS / BLOCKING CONCERN) so all transform reasoning dies with the worker.
- **`groundwork-doc-sync/instructions.md`** — point the Living Documents protocol at a code change set: establish the range, map it to affected docs in three passes (path intersection → reference graph → semantic judgment), edit surgically, classify refinements vs reversals, gate mutated canonical docs, commit once on approval.
- **`groundwork-surface-activation/instructions.md`** — make surface divergence a recorded decision, never drift: register the surface, reuse/lazily-run the type design track, scaffold (or record `manual` obligations), and — the skill's reason to exist — triage every capability-ledger row so the new column has no empty cell, cross-posting `planned` cells into discovery notes as bet backlog. Plus a registry bootstrap for products that predate the surface model.
- **`migrations/README.md`** — keep contributors on the right side of the mechanical/judgment boundary and make every registry entry conform to the contract the gate tests enforce (forward-only, detect-first, idempotent, self-contained).
- **`migrations/index.json` + `_template/` + `gw-*.js`** — the shipped mechanical lane itself; the template is the authoring on-ramp that restates the contract at the point of use.

## 3. Distillation moves

**M1 — One driver loop, stated once** (MAINT-8, MAINT-1, MAINT-15, MAINT-12). Restructure `groundwork-update/instructions.md`:
- Keep the intro's two-lane mental model (brief items / structural drift) — it is the front-loaded framing skill-writer asks for.
- "You are the driver" keeps the isolation rationale (the driver makes the dispatch decision); it becomes the *only* home for it.
- Move `fan_out` and phase ordering out of "## Operating Contract" into the driver section; compress `fan_out` to the branch behavior (`parallel` → dispatch at `execution` tier; `sequential` → inline one unit then purge; no hint → `sequential`), one sentence of why.
- New "## The driver loop" defines the shared machinery once: **propose (one unit, never batch) → dispatch or inline → resolve COLLISIONS/AMBIGUITY with the user → gate every REVIEW-NEEDED doc (Protocol 9, fail-closed, 3-revise cap) → stage + commit (one unit, one commit, message format) → record (Phase A only)**.
- Phase A shrinks to: brief location/shape, top-to-bottom order, per-type capsule pointers, the Bookkeeping contract verbatim (MAINT-17 — untouched).
- Phase B shrinks to: detect-cheaply-in-the-driver guidance, the pause-on-implied-product-decision rule, and the Family Index.
Before: two 6/4-step loops that must be read against each other. After: one loop the driver internalizes, two short phase deltas — better pacing and gating because the loop is unambiguous, not because it is shorter.

**M2 — Recipes live only in the worker brief** (MAINT-2, MAINT-5). Reshape the Family Index into three columns: *Family (+ slug)* | *Owner — current canonical* | *Legacy signal* | *Advance* — where *Advance* is a recipe name from the worker brief plus family-specific parameters only (e.g. Bets: `restructure — in-flight only; a pitch with track: quick is a legitimate single-milestone bet, leave it intact`; Naming becomes three signal bullets each naming `rename` or `reference-rewrite`). The full transforms (graduate-and-stamp mechanics, reference-rewrite scope rules, regenerate steps) stay solely in `reconcile-worker.md`, which the sequential-mode driver already reads inline by instruction. This closes the internal contradiction and makes the index what the driver needs it to be: a fast detection table. Adding the slug column also settles MAINT-16 (the worker's `family:<name>` examples become derived, not parallel).

**M3 — Enact Protocol 1 in update** (MAINT-3). One close-out line in the driver loop's final section: stray signals voiced during the catch-up go under their headers in `.groundwork/cache/discovery-notes.md` (Protocol 1) — mirroring doc-sync Step 5.2. Cheaper than the alternative (dropping the P1 claim) and correct: a version catch-up conversation does surface product signals.

**M4 — Worker brief sheds the driver's argument** (MAINT-1, MAINT-7). `reconcile-worker.md` "How This Brief Is Invoked" compresses to: runs isolated, dispatched once per unit by the groundwork-update driver, capsule in / report out, reasoning dies here. The "Invocation environments" table becomes one sentence deferring to the host's subagent mechanism (Protocol 9 owns dispatch mechanics); "### Model" becomes one sentence (frontmatter `tier: execution` + "gated at frontier by the driver" once). The recipes section and the report format are untouched — they are the brief's value.

**M5 — Single boundary statement in migrations/README** (MAINT-4). Keep the ¶2 statement (it is the stronger one — names the Family Index and the "no per-change migration" rule); the post-checklist paragraph becomes one pointer sentence; the template header comment keeps its one-line redirect (it is the point-of-use guard for someone who skipped the README) but drops the restated rationale.

**M6 — Compress the bootstrap, stage its retirement** (MAINT-6). Now: cut the bootstrap section ~340→~200 words (the reconstruction sources, the empty-ledger honesty rule, and the confirm-before-write step are the intent; the rest is elaboration). Add to the master plan an evidence task: check npm version history + known pre-publish installs; on confirmation, retire the section plus its two external references (orchestrator routing clause, Surfaces-registry family's bootstrap arm) in one change — they form a reference cycle that must move together.

**M7 — Delegate track-standalone mechanics to the design-system tracks** (MAINT-10). Surface-activation Step 2 keeps the decision logic (type section exists → point and move on; missing → run the type's track against the committed foundation; brand conversation never re-runs) and drops the internals (Phase 5/Phase 6 numbering, cache-file substitution). The "how to run a track standalone" note moves to where the tracks are owned — see FLAG(3). Until that lands, Step 2 keeps a single pointer to the track file rather than paraphrasing its phases.

**M8 — doc-sync trims** (MAINT-11, MAINT-14). Step 1's no-anchor row becomes "run `groundwork-check`'s baseline procedure (its SKILL.md defines it); the union of flagged commits is the change set" — one owner for the procedure. The surgical-intent statement stays in Step 3 (where it calibrates) and drops from the opening paragraph's second restatement. doc-sync is otherwise the healthiest file in the section: three-pass mapping, refinement/reversal classification, and the semantic-mapping table are exactly the calibration agents need — no further cuts.

**M9 — index.json summaries to one line each** (MAINT-13). Contributor-facing only; the module header comment is the full story. Gate-safe (asserts non-empty).

## 5. Risks & cross-section flags

Execution risks:

- **Writers/readers of shared identifiers this plan touches**: Family Index family names/slugs (writer: `groundwork-update` instructions; readers: `reconcile-worker` capsule examples, `migrations/README.md`, contributor guide tier-4 row, `gw-bet-prose-redesign` summary — M2's slug column must land in all in one change). Brief item fields (`tier2-merge`/`tier1-custom`/`regenerate`, `incoming`, `base_hash`) are written by `bin/groundwork.js` and read by both skill files — M1/M4 rewording must not touch these strings. `track: quick`, `bet/<slug>/approved`, `## Bets` discovery-notes header, `docs/surfaces.md`/`.groundwork/surfaces.json` — all read here, written elsewhere (see flags).
- **Operating-contract conformance**: the contract's Maintenance mode paragraphs (operating-contract.md "### Maintenance (anytime)") describe each of these skills' protocol sets and *additional obligations* — any restructure that renames Phase A/Phase B or the registry-twins obligation must update that contract paragraph in the same change, or the contract and skill describe different skills.
- **Shipped-surface changes**: every file in this section ships (tier 1 clean-replace for the skills; `migrations/` rides the package). Skill rewrites are clean-copy exempt from migrations, but the CLI's console message at `bin/groundwork.js:1453` names "the groundwork-update skill consumes it, then reconciles structure" — keep the two-phase vocabulary or update the string. Changelog lines for the rewrites are `[no-migration]`.
- **M2 depends on the driver actually passing the row's parameters in the capsule** — the capsule contract ("Advance approach" bullet) already requires this; the rewrite must keep that bullet explicit or quick-bet protection silently weakens.

FLAG(1): orchestrator SKILL.md routing rows for `groundwork-update` (L104), `groundwork-doc-sync` (L103), `groundwork-surface-activation` (L108) restate these skills' internals (Phase A/B family list, bootstrap clause) — if this section's restructure or the bootstrap retirement (M6) lands, those rows move in the same change. FLAG(4): brownfield — the bootstrap-retirement evidence (M6/MAINT-6) partly rests on `groundwork-architecture-extract` writing the surfaces registry at every brownfield commit; confirm that section keeps this unconditional when distilling extract.
FLAG(5): bet-delivery — the Bets family reads `track: quick` frontmatter, the `bet/<slug>/approved` tag, and the prose `decomposition/` tree shape as legacy/current signals; if bet-delivery distillation renames any of these, the Family Index row and `gw-bet-prose-redesign`'s summary are readers to update in the same change. FLAG(10): groundwork-contributor "Writer Enforcement" table is stale for this section — "Update ✅ Top-level mandate" (the writer reference actually lives in `reconcile-worker.md`, which is the correct placement and should be recorded as such) and a "Docs Uplift" row for a skill that no longer exists. Fix the table, not the skills.
FLAG(10): skill-writer standard held up well as a lens here (its "shared-contract non-conformance" and "identifier drift" failure modes found MAINT-3 and MAINT-16 directly). One gap: it has no named failure mode for **driver/worker instruction pairs** (rationale belongs with the decision-maker; recipes belong with the executor; each stated once) — MAINT-1/MAINT-2 are that pattern, and bet-delivery's slice-worker pair will hit it too.
