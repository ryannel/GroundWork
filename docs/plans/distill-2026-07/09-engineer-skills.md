# Distillation Plan — Engineer Skills

Status: PROPOSED (input to the master distillation plan)
Scope: 69,237 words total (`wc -w` over all `.md`): go-engineer 10,816 · python-engineer 9,901 · nextjs-engineer 24,474 · flutter-engineer 13,499 · electron-engineer 10,547

## 1. Intention

An engineer skill is promoted into a scaffolded project's `.agents/skills/` (`promoteEngineerSkill`, `src/generators/shared/scaffold-helpers.ts`) and becomes the *only* registered per-language skill in that project — its description is an always-on context tax on every session, and its body loads on every coding task in that language. Its job is not to teach the model the framework (the model already knows Go, FastAPI, React); it is to defeat **plausible-but-wrong output** by doing three things the model cannot do from training data:

1. **Pin the project's conventions** — the scaffold's actual layout, generated files, token projection chain, contract locations — so the agent verifies instead of inventing (`Hallucination Controls`, `Required First Checks`).
2. **Calibrate general knowledge toward the stack's opinionated choices** — Riverpod-not-GetX, SWR-not-React-Query, declarative-schema-not-Alembic-scripts, honeycomb-not-pyramid — with just enough reasoning that the agent can defend the default under pushback (skill-writer: "opinionated defaults need reasoning").
3. **Correct where training data is stale** — Next.js 15 async params, Electron fuses post-CVE-2025-55305, Flutter 3.44 SwiftPM default. These are the highest-value words in the section: without them the model confidently writes yesterday's idiom.

The two-family split is load-bearing and correct. **Backend** (go, python) work is gated by specs and sibling services, so the spine front-loads contract checks, handoffs, and an execution checklist. **Surface** (nextjs, flutter, electron) work fails as *plausible-but-wrong UI idiom* — wrong boundary, hardcoded visual literal, re-proven core rule — so the spine front-loads pillars, hallucination controls, and antipatterns. Nothing in this plan changes the family spines.

Per-skill intention map (references grouped):

| Skill | SKILL.md intent | references/ intent |
|---|---|---|
| go | Route to minimal references; safety gates on contract/layer/concurrency damage | Pure-core layout + Go idiom (services, concurrency, handlers, postgres, capability ports); distilled canon (testing, observability, reliability/perf, security/craft, docs) |
| python | Same, plus ML-pipeline gates | Same shape plus async/lifespan, FastAPI defenses, ML/MCP idiom |
| nextjs | Boundary decision (server/client), token consumption, a11y as gate | App Router boundary + data/mutation idiom; design-system consumption technique; testing trophy; version-gotcha corrections |
| flutter | MVVM/Riverpod/theme-projection discipline; prove-once testing | Official-architecture idiom per concern; store/release engineering; canon distills |
| electron | Process/IPC/security discipline; the `_electron` smoke loop | Hardened process model, IPC contract, fuses/signing; explicit deferral of web-stack knowledge to the nextjs skill |
| all | `sync-anchor.md` = repo-side CI pin of the distilled principle sources | — |

The quality benchmark already exists inside the section: the flutter and electron references (and python's security.md) are decision-time distillations — opinionated, canon-linked, every example calibrating. The distillation should converge the rest of the section on *that* genre, not invent a new one.

## 5. Risks & cross-section flags

**Execution risks**

- **Sync-anchor cascade.** M2/M3/M4 touch files whose sources are pinned. Editing a reference does not break the gate (hashes pin the *sources*), but any pass that also edits a `src/docs/principles/*` or stack doc must re-stamp anchors in the same commit — and per ENG-2, a re-stamp without a real content review is the exact anti-pattern this section documents. Rule for executors: a hash bump in `sync-anchor.md` requires a sentence in the commit naming what was reconciled.
- **Promotion-mechanism change (M6, and M4's fallback).** `promoteEngineerSkill` is generator code: needs generation-test updates (`tests/scaffolds/test_generation.py` expectations for promoted trees) and a changelog line. `[no-migration]` is defensible for M6 (generative path only; stale installs keep an inert file). The M4 fallback (`_shared/` copied into references/) is a bigger change — only take it if the master plan rejects canon-deferral.
- **Propagation gap.** `groundwork update` deliberately *preserves* promoted engineer skills (`ownedRegisteredSkillNames` in `bin/groundwork.js` excludes `src/engineer-skills/`; the update-skill-deletion fix made this a rule). All content improvements here reach only newly scaffolded projects. See FLAG(6).
- **Template-family divergence policy.** M1/M8 restructure nextjs references while electron *defers into them by path* (SKILL.md deferral block). Every rename/section-move must update electron's deferrals in the same commit (writer/reader rule), and flutter/electron must be re-checked for any incidental reliance on nextjs file names.
- **Cut-too-deep risk on ENG-1.** Some tutorial-looking content doubles as project convention (SWR-fallback hydration seeding, MSW-not-fetch-mock, the API-client layer shape). The move keeps anything that pins a *choice*; executors should cut only what pins no choice and corrects no staleness.

**Cross-section flags**

FLAG(6): `groundwork update` never refreshes promoted engineer skills (`ownedRegisteredSkillNames` excludes `src/engineer-skills/`; they are preserved byte-identical). This distillation will not reach any existing install — maintenance/update needs a propagation path, e.g. a reconcile family or provenance-based re-promotion (`manifest.generated` already records them) in the groundwork-update Family Index.
FLAG(8): go `code-craft-security.md`, `reliability-performance.md`, `observability.md`, and `integration-realtime-data.md` are near-copies of principles-corpus headings — check corpus-side whether `quality/*` and `foundations/*` are written to be *deferred to* by skills (stable paths, self-sufficient summaries) rather than re-stated; and audit generator-shipped `docs/principles/stack/<lang>/` vs skill references for the same near-copy pairs (ENG-16).
FLAG(8): the sync-anchor gate verifies hashes only, so anchors can be re-stamped without content realignment (ENG-2 is a live instance; the principles-rewrite memory records another). Consider a gate or convention requiring a reconciliation note per re-stamp.

Propose contributor/skill-writer add: references are decision-time distillations (pin conventions, calibrate defaults, correct staleness — never restate framework manuals), and extend "copy the newest sibling from the same family" explicitly to reference files.
