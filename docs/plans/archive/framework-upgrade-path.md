# Implementation Plan: The Framework Upgrade Path (No Project Left Behind)

**Status:** EXECUTED 2026-06-12 — all workstreams A–H landed. Remaining: the H3 live
simulation run (suite seeded at `tests/evals/scenarios/upgrade/`; needs a human-driven
session per the flow-testing harness) and the catch-up gate's live `groundwork-upgrade`
session (same dependency). G1's tag-diff gate is committed but skips until the first
release tag exists. See Execution notes (§13) for deviations.
**Audience:** An engineer or agent implementing this change. Each slice names its files and an acceptance check; judgment calls that remain are listed as open decisions in §9.
**Scope owner:** `bin/groundwork.js` (`update`/`check`), a new `migrations/` registry in the package, a new hidden skill `groundwork-upgrade`, the `workspace-dev-cli` generator, with coupled changes to `groundwork-check`, the orchestrator, the contributor guide, and the release workflow.

## 0. Read this first — the mental model

A GroundWork install is not a snapshot. It is a **deployment of the framework into a project**, and a deployment that cannot track its source strands every project at the version it happened to adopt. Today exactly one tier of the install tracks the framework: skills, which `npx groundwork-method update` clean-copies. Everything else — the principles docs seeded into `docs/`, the `./dev` CLI generated by scaffold, the `state.json`/`surfaces.json` schemas, the canonical doc shapes — is written once and never carried forward. When we improve how the dev CLI works, or rewrite a principle, or rename `docs/ux-design.md` to `docs/design-system.md`, existing projects keep the old world. The only bridge is a `[migration]` line in the changelog that a human may or may not read, and that nothing tracks the completion of.

**The organizing idea: every installed artifact gets an owner and a provenance record, and every framework change that touches an installed artifact ships with a migration that knows how to carry that artifact forward.** The work splits along the line this repo already draws — *the CLI's job is file copying and user messaging; methodology and intelligence belong in skills*:

- **The mechanical lane (CLI).** Deterministic, hash-verified operations: replace framework-owned files, refresh pristine seeded docs, run scripted migrations, record completion. Runs inside `groundwork update`, no agent in the loop.
- **The judgment lane (skill).** A new hidden skill, `groundwork-upgrade`, walks the project through everything that needs eyes: merging framework improvements into user-edited docs, restructuring canonical documents to new shapes, regenerating scaffold output and reconciling the diff. The CLI compiles its work list (the *upgrade brief*); the skill executes it conversationally, one commit per step.

The enabling structure is an **ownership model** for everything GroundWork puts into a project:

| Tier | What | Examples | Carry-forward rule |
|---|---|---|---|
| 1 | Framework-owned, never user-edited | `.agents/skills/`, `.agents/groundwork/skills/`, `generators.json`, agent symlinks, `.dev/dev-bundle.js` + `dev` launcher | Clean-replace on `update` (skills already work this way; the dev bundle is the headline addition) |
| 2 | Framework-seeded, user-editable | `docs/principles/`, `docs/ways-of-working/`, `AGENTS.md`, `llms.txt` | Hash-stamped at deploy. Pristine → auto-refresh; edited → queued for skill-mediated merge |
| 3 | Generator-produced, options-parameterized | `docker-compose.yml` injections, `tests/system/` runner, `.dev/dev.config.json`, `workspace-cli` skill, service operational layer | Provenance-recorded (generator, version, options). Upgrade regenerates with recorded options and reconciles the diff |
| 4 | Project-owned, shape-versioned | `state.json`, `surfaces.json`, `decomposition.json`, doc frontmatter conventions, canonical doc set shape | Never overwritten. Versioned migrations transform them in place |

Tiers 1–2 are CLI work. Tier 3 is CLI-detected, skill-reconciled. Tier 4 is migrations — scripted when mechanical (a JSON schema bump), brief-driven when semantic (a doc rename, a new required section, a registry bootstrap).

What this plan is **not**: it is not Nx's `migrations.json` (our migration surface is mostly docs and skills, not TypeScript config — see settled decision S5), and it is not a sync daemon. The cadence stays explicit: the user (or their agent, prompted by `groundwork check`) runs `update`, then the upgrade skill finishes what file copying can't.

## 1. Findings this plan responds to

| ID | Finding | Severity |
|---|---|---|
| U1 | `update` refreshes only skills + `generators.json`; seeded docs, generator output, and doc shapes have no carry-forward path at all | High — core gap |
| U2 | `[migration]` changelog lines are advisory prose; nothing is machine-actionable, nothing records whether a migration was performed, repeated `update` runs re-show nothing | High — core gap |
| U3 | Tier-2 docs are copy-if-absent (`copyDocsIdempotent`, `bin/groundwork.js`): a principles rewrite never reaches any existing install, and we cannot tell an intentional user edit from staleness | High |
| U4 | No generator provenance: a project cannot know which generator version/options produced its `./dev`, compose file, or test runner. Dev-CLI improvements (`cli-src/` → rebuilt bundle) reach only *newly scaffolded* projects | High — the user-named example |
| U5 | Structural schemas carry version fields (`state.json` `version: 2`, `surfaces.json` schema 1) but no runner transforms old shapes when the schema moves | Medium-High |
| U6 | Semantic doc-shape changes have no general mechanism. Each restructure invents a one-off: surface-activation bootstraps the registry on pre-restructure products; the design-system rename shipped with no uplift at all — old projects still have `docs/ux-design.md` | High |
| U7 | Pre-0.9 installs are unstamped until their first `update`; version detection has a fallback but nothing else does | Low |

| U9 | No contributor forcing function: a PR can change `src/docs/`, generator templates, or a schema without shipping a migration, and nothing objects | Medium |

**Strengths to preserve:** the kill-and-replace skill update (never merge framework-owned instruction files); diff-before-copy reporting; the preserve-list discipline (`state.json`, `config.toml`, `docs/`, cache); the three-version-point release rule (package / stamp / contract); the CLI-simple-skills-smart split.

## 2. Workstream A — Install manifest & provenance (the foundation)

Nothing downstream works without knowing *what was deployed, from which version, and whether the user touched it*. This is the upgrade path's equivalent of the bet loop's lock manifest.

**A1 — The install manifest.**
`update`/`init` write `.groundwork/config/manifest.json`: for every tier-1 and tier-2 file deployed — relative path, tier, source path in the package, package version at deploy, SHA-256 of the deployed content. Schema documented in the file header (`manifest_version: 1`). `state.json` stays pure lifecycle state; the manifest is a separate file (decision S2). *Files:* `bin/groundwork.js` (new `writeManifest()`, called from init + update). *Accept:* fresh init produces a manifest covering every file the installer wrote; entries verify against disk.

**A2 — Generator provenance.**
Every generator records what it produced: `workspace-dev-cli`, `system-test-runner`, and the app/service generators append to a `generated` section of the manifest — generator name, package version, resolved options, file list with hashes. The scaffold skill already invokes generators through `generators.json`; the recording hook lives in a shared helper the generator factories call. *Files:* `src/generators/*/generator.ts` (or the shared util they import), manifest schema. *Accept:* scaffolding in a sandbox yields manifest entries for `.dev/`, compose, `tests/system/` naming the generator + options that produced them.

**A3 — Manifest bootstrap for existing installs.**
On `update` against a project with no manifest: backfill it. For each expected tier-1/2 file, hash what's on disk; if it matches the *current package's* content, record it pristine; otherwise record `provenance: "adopted"` (unknown ancestry — treated as user-edited until proven otherwise). Generator output without provenance is recorded `adopted` with a detection heuristic (`.dev/dev-bundle.js` exists → workspace-dev-cli ran). *Accept:* running `update` on a pre-manifest fixture produces a complete manifest with correct pristine/adopted classification.

**A4 — Manifest contract tests.**
Table-driven tests in `tests/cli/`: manifest completeness after init, hash verification, bootstrap classification, manifest survival across repeated updates. *Accept:* green in `./dev test contracts` (or the `cli` lane if A3 of contract-grade-delivery landed one).

## 3. Workstream B — The migration registry (mechanical lane)

**The shift:** `[migration]` stops being prose and becomes a registry entry with code or a brief attached. The changelog line remains — for humans — but it names a migration ID the tooling can cross-check.

**B1 — Registry format.**
New top-level `migrations/` directory in the package, shipped via `package.json` `files`. `migrations/index.json` lists entries: `{ id, version, title, kind: "cli" | "agent", summary }`. Each `cli` migration is a single CommonJS module `migrations/<id>.js` exporting `detect(project) → "pending" | "done" | "n/a"` and `run(project)`; each `agent` migration is a directory `migrations/<id>/brief.md` (consumed by WS-E). Rules: forward-only, idempotent, detect-first — `run` is only called when `detect` says pending, and running twice must be a no-op. *Accept:* format documented in `migrations/README.md`; index validates against a schema test.

**B2 — `update` runs pending CLI migrations.**
After the skill refresh and before stamping, `updateGroundWork()` loads the index, filters entries with `version > stampedVersion` (plus any whose `detect` reports pending regardless of stamp — repairs missed runs), executes `cli` migrations in version order, and records completions in `state.json` under `groundwork.migrations: ["<id>", …]`. Failures stop the run with the migration named; the stamp is not advanced past a failed migration's version. *Files:* `bin/groundwork.js`. *Accept:* a fixture with an old stamp gets exactly the pending migrations run, recorded, and skipped on the next update.

**B3 — Changelog ↔ registry cross-check.**
A `[migration]` changelog line must end with its registry ID in parens — `[migration] … (gw-0010-dev-bundle-refresh)`. A contract test parses `CHANGELOG.md` and `migrations/index.json` both ways: every `[migration]` line names a real entry; every entry at a released version has a changelog line. `printChangelogSlice` keeps working unchanged. *Accept:* test red when either side is missing.

**B4 — `update --dry-run`.**
Prints the full plan without writing: skill diff (already computed), tier-2 refresh/merge classification, pending migrations by kind, and what the upgrade brief would contain. This is also what `groundwork check` links to (WS-F). *Accept:* dry-run on a fixture mutates nothing (verified by hashing the tree before/after) and lists the same actions a real run performs.

## 4. Workstream C — Tier-2 refresh: seeded docs stop fossilizing (U3)

**C1 — Pristine/edited/absent classification.**
`update` classifies every tier-2 file: disk hash == manifest deploy hash → **pristine** (the user never touched it) → replace with the current package content and re-stamp the manifest, listing the refresh in the diff report. Differs → **edited** → leave untouched, queue into the upgrade brief with the package's new content for a skill-mediated merge. Absent → copy (current behavior). `adopted` entries from A3 are treated as edited. *Files:* `bin/groundwork.js` (`copyDocsIdempotent` becomes `refreshTier2()`). *Accept:* fixture with one pristine, one edited, one missing principles file → refresh, queue, copy respectively.

**C2 — `AGENTS.md` and `llms.txt` join tier 2.**
Same classification. `AGENTS.md` is the highest-value target — it routes every agent session, and today an init-day copy never improves. *Accept:* covered by C1's test matrix.

**C3 — `config.toml` stays user-owned, gains additive healing only.**
Never refreshed or merged. If a new release adds a *required* key, that is a `cli` migration that inserts the key with its default and a comment — never touching existing keys. *Accept:* a seeded migration adds a key to a fixture config.toml without reordering or reformatting the rest.

## 5. Workstream D — Tier 1 grows: the `./dev` CLI tracks the framework (U4)

The user-named example, and the cleanest win in the plan. The dev bundle is *designed* to be framework-owned — zero-dependency, built from `cli-src/`, copied verbatim, never meant to be user-edited. It only lives in tier 3 today because nothing distinguishes it from genuinely project-shaped generator output.

**D1 — Bundle + launcher become tier 1.**
`update` detects a project with `.dev/dev-bundle.js` (manifest provenance, or A3 heuristic) and clean-replaces the bundle and the `dev` launcher from the package's committed `cli-src/dist/dev-bundle.js` + `dev.template`, hash-verified, listed in the diff report. `.dev/dev.config.json` is **not** touched — it is projected from the project's brand tokens (tier 3). If the launcher hash matches neither the manifest nor any shipped version, warn and skip (someone customized it; queue to the brief). *Files:* `bin/groundwork.js`; `package.json` `files` must ship the bundle (verify — the generator needs it at scaffold time already). *Accept:* an old-bundle fixture gets the current bundle on `update`; `./dev --version` (D3) reports the package version.

**D1.5 — The project grows the toolkit beside the framework core (added 2026-06-21, plan: `customization-and-forge.md`).**
Clean-replacing the bundle is right for the *core* but must never erase what the project added. So the `./dev` toolkit splits: the bundle + launcher stay tier-1, clean-replaced (framework plumbing the user does not edit), while a **project-owned command layer** — JSON commands under `.dev/commands/` and a `commands` block in `.dev/dev.config.json` — is the project's own. The bundle discovers and runs those commands at runtime (they can even shadow a built-in), and `update` never touches them: `.dev/commands/` is in no manifest tier, and the generator preserves the config `commands` block on regenerate exactly as it preserves `runners`. This is how "the `./dev` set is heavily customised as the app is built out" coexists with a clean-replaced core — the user's customisation lives outside the artifact that gets replaced. Framework *capability* improvements to `./dev` (new ideas, not just new bytes) ride the agent-brief path (WS-E): the brief carries the thinking and how it was applied to GroundWork's reference `./dev`, and the upgrade skill works the equivalent into the project's grown toolkit rather than overwriting it.

**D2 — Tier-3 regenerate-and-reconcile.**
For provenance-recorded generator output that *is* project-shaped (compose injections, `tests/system/` runner, `workspace-cli` SKILL.md): the upgrade brief lists each artifact whose generator changed since deploy. The upgrade skill re-runs the generator with the recorded options into `.groundwork/cache/upgrade/<artifact>/`, diffs against the project, and walks the user through adopting hunks — framework-section changes proposed, user-section changes left alone. Requires A2. *Files:* `groundwork-upgrade` skill (WS-E), generator dry-run support if absent (contract-grade F4 proposed `--dry-run`). *Accept:* a sandbox whose system-test runner predates a template change gets a correct proposed diff.

**D3 — The bundle knows its version.**
Embed the package version in the bundle at build time (`npm run build:dev-cli` injects it); `./dev --version` prints it; `./dev doctor` warns when it trails the installed skills' stamp. *Files:* `src/generators/workspace-dev-cli/cli-src/`, build script. *Accept:* doctor flags a deliberately downgraded bundle.

## 6. Workstream E — The `groundwork-upgrade` skill (judgment lane)

**E1 — Skill skeleton.**
New hidden skill `src/hidden-skills/groundwork-upgrade/instructions.md` (contract v1 reference, writer reference, orchestrator routing row — the standard checklist from the contributor guide). Its input is the **upgrade brief**: `.groundwork/cache/upgrade-brief.json`, written by `update` whenever judgment-lane work exists — pending `agent` migrations (version-ordered), edited tier-2 files queued by C1 (with the new package content inlined or referenced), tier-3 reconciliation candidates from D2, and the version jump (`from`, `to`). The skill walks the brief top to bottom: one item → one explained proposal → user approval → one commit. Completed items are checked off in the brief; the brief is deleted when empty (cache lifecycle rules apply). Naming: `groundwork-update` already exists for *document* updates — the routing descriptions must keep the pair unambiguous (see settled decision S6). *Accept:* `npx groundwork-method init` in a test repo installs the skill; the orchestrator routes "upgrade groundwork" / "bring this project up to date" to it.

**E2 — Agent-migration briefs.**
The format for `kind: "agent"` registry entries: `migrations/<id>/brief.md` with three required sections — **Detect** (filesystem/content conditions that mean the migration applies; also how to recognize it's already done), **Transform** (the change, written as intent + invariants, not a script), **Accept** (what a correct end state looks like). The upgrade skill loads the brief verbatim as its instructions for that item. *Accept:* brief template committed; E3's backfilled briefs conform.

**E3 — Backfill briefs for the already-shipped restructures.**
The framework has already left projects behind; these briefs are the catch-up path that makes the mechanism honest on day one:

1. `gw-design-system-rename` — detect `docs/ux-design.md`; rename to `docs/design-system.md`, update cross-references in docs and state.
2. `gw-surfaces-registry-bootstrap` — detect canonical docs without `docs/surfaces.md` / `.groundwork/surfaces.json`; delegate to `groundwork-surface-activation`'s existing bootstrap flow rather than duplicating it.
3. `gw-drift-frontmatter-stamp` — detect canonical docs lacking `last_reviewed`/`source_of_truth`; stamp them so `groundwork check` becomes effective (the brownfield-extract phases already define the stamping rules; reference, don't restate).
4. `gw-bet-shape-uplift` — detect pre-restructure bet directories (no `decomposition.json`, old pitch frontmatter); uplift in-flight bets' tracking files, leave archived bets alone.

*Accept:* each brief proven against a fixture seeded from the corresponding pre-restructure shape (H1 fixtures).

**E4 — Post-update handoff.**
`update`'s closing output, when a brief exists: one line — *"N items need a working session: open your agent and say 'upgrade groundwork'."* The orchestrator's session-start reconciliation surfaces the same line whenever a brief file exists. *Accept:* simulation transcript shows the handoff line and the skill picking the brief up in a fresh session.

## 7. Workstream F — Staleness surfacing: the project knows it's behind (U10)

**F1 — `groundwork check` gains a framework section.**
Before the doc-drift report: installed stamp vs package version (the package running the check *is* the latest the user fetched — no network call; see open decision O2), pending migrations by `detect()`, tier-1 hash mismatches (a user-edited skill file is corruption, name it), unconsumed upgrade brief. Exit code stays drift-semantics; framework staleness adds a distinct line and exit code 1. *Accept:* check on an old fixture names the version gap and pending migration count.

**F2 — `groundwork-check` skill mirrors it.**
The registered check skill's instructions cover reading the new section and routing: mechanical staleness → tell the user to run `npx groundwork-method update`; brief exists → route to `groundwork-upgrade`. *Accept:* skill text updated; routing verified in a sandbox session.

**F3 — `./dev doctor` joins in.**
D3's bundle-version check plus a pointer to `groundwork check` when the stamp trails. *Accept:* covered by D3.

## 8. Workstream G — Contributor forcing functions (U9) + release wiring

**G1 — The migration-coverage gate.**
A contract test (in `./dev test contracts`, so it runs in the release workflow's cheap gates): diff the working tree's *shipped surface* against the last released tag — `src/docs/`, `src/config/`, `src/generators/*/files/`, `cli-src/dist/`, schema-bearing files (`groundwork-state.json`, surfaces schema) — and require that any change is matched by either a migration entry at the unreleased version or an explicit `[no-migration]` annotation on the changelog line (for additive-only changes that old projects don't need). Skills are exempt — clean-copy already carries them. *Accept:* touching a principles doc without a migration or annotation turns the gate red.

**G2 — Contributor guide + release checklist.**
`groundwork-contributor` SKILL.md gains a *Shipping a change that touches installed projects* section: the tier table from §0, when to write `cli` vs `agent` migrations, the brief format, and the registry ID convention. The release checklist adds: run G1's gate, verify new migrations have fixtures (H1). *Accept:* sections committed; checklist references the gate.

**G3 — Migration authoring template.**
`migrations/_template/` with a skeleton `brief.md` and a skeleton `cli` module, referenced from G2. *Accept:* exists; E3 briefs were authored from it.

## 9. Workstream H — The upgrade path is itself tested (U8)

**H1 — Old-install fixtures.**
`tests/fixtures/installs/<label>/` — committed snapshots of project trees as older GroundWork versions left them: `pre-0.9` (unstamped, pre-rename), `0.9-pre-surfaces` (stamped, old doc shapes, old dev bundle), plus a synthetic `edited` variant of each (user modifications to tier-2 docs and one test file). Built once from git history (check out the tag, run init into a temp dir, commit the result as a fixture) and frozen — fixtures are inputs, not goldens to re-harvest. Each future release that ships a migration adds the fixture state it migrates *from* if no existing fixture covers it. *Accept:* fixtures committed with a README stating provenance (tag + command).

**H2 — Upgrade contract tests.**
The core invariants, table-driven against H1 fixtures in `tests/cli/`:
- **Convergence:** after `update`, every tier-1 file is byte-identical to a fresh init at the current version.
- **Preservation:** user-edited tier-2 files, `config.toml`, `docs/` content, `.groundwork` state survive byte-identical (modulo the stamp and recorded migrations).
- **Idempotency:** a second `update` is a no-op (tree hash unchanged, no migrations re-run).
- **Detect-honesty:** every registered migration's `detect` returns `done` or `n/a` on a fresh init.
*Accept:* green in `./dev test contracts`; wired into the release workflow's gates.

**H3 — Upgrade simulation suite.**
`tests/evals/scenarios/upgrade/` — persona: a developer returning to a project last touched several framework versions ago, mildly protective of their edited principles doc. Seeder starts from an H1 `edited` fixture, runs real `update`, then the session runs `groundwork-upgrade` through the brief. Judge criteria: edited-doc merge preserved the user's intent, every brief item ended in a commit, nothing outside the brief was touched. *Accept:* suite seeded and run once live; transcript + judge report referenced from this plan.

## 10. Decisions

### Settled

| # | Decision | Rationale |
|---|---|---|
| S1 | Mechanical/judgment split: CLI copies and verifies, skill merges and restructures | The repo's standing rule — "the CLI's job is file copying and user messaging; methodology and intelligence belong in skills" — applied to upgrades |
| S2 | Provenance lives in a dedicated `manifest.json`, not `state.json` | state.json is lifecycle state read by the orchestrator every session; a file-hash ledger is operational metadata with a different reader and growth rate |
| S3 | Hash manifest over git-based edit detection | User projects' git history is theirs — rebased, squashed, or absent; a self-contained hash record needs no assumptions about it |
| S4 | Migrations are forward-only, idempotent, detect-first; no down-migrations | Projects do not downgrade frameworks; idempotency + detect make re-runs and missed-stamp repairs safe, which is cheaper than rollback machinery |
| S5 | Own registry, not Nx `migrations.json` | Our migration surface is mostly docs, skills, and shapes — not workspace config; Nx's runner assumes package.json-versioned Nx plugins and brings none of the agent-brief lane |
| S6 | The skill is named `groundwork-upgrade`; `groundwork-update` keeps its name | Renaming the existing doc-update skill would itself strand old routing. Disambiguation lives where routing happens: the orchestrator table describes one as "update project documents" and the other as "bring the project up to the current framework version" |
| S7 | The dev bundle + launcher are tier 1 (framework-owned) | The bundle is built, zero-dependency, and copied verbatim by design; treating it as project code is what stranded it. Anyone who truly forks it fails the hash check and lands safely in the judgment lane |

### Open

| # | Decision | Options | Default proposal |
|---|---|---|---|
| O1 | Tier-2 pristine refresh: automatic or always previewed? | auto-replace with diff line / prompt per file | Auto-replace — pristine means the user delegated the file to the framework; the diff report keeps it visible, `--dry-run` exists for the cautious |
| O2 | Network staleness check (`check` queries npm for newer versions)? | yes, best-effort with cache / no — offline only | No for now. `npx groundwork-method@latest check` already fetches the latest package; an HTTP call inside `check` adds flakiness for a marginal signal |
| O3 | Backfill depth for E3 | the four named briefs / everything since 0.8 changelog archaeology | The four named briefs — they cover the restructures real projects exist on; older gaps get briefs on demand when a fixture proves them needed |
| O4 | Should `update` auto-launch the upgrade skill? | print handoff line / attempt to open an agent session | Handoff line only (E4). The CLI must not assume which agent is installed or that a TTY can host one |
| O5 | Tier-2 merge mechanics in the skill | skill reasons over both versions / CLI pre-computes a 3-way merge with conflict markers | Skill reasons over both versions with the manifest's base hash available — 3-way text merge produces marker soup in prose docs, which is exactly the judgment lane's job to avoid |

## 11. Sequencing

```
WS-A (A1→A2→A3→A4)            ──► the foundation; blocks everything below
WS-B (B1→B2, B3–B4 parallel)  ──► blocks C (queueing), E (brief), F (pending counts)
WS-C (C1→C2→C3)               ──► after A+B; C1 feeds the brief format
WS-D (D1, D3 anytime after A; D2 after E1)
WS-E (E1→E2→E3→E4)            ──► E1 needs B's brief writer; E3 needs H1 fixtures to verify
WS-F (F1→F2→F3)               ──► after B2 (pending detection); cheap, ship early otherwise
WS-G (G1–G3)                  ──► G1 lands with B1 so the gate exists before the next release
WS-H (H1 early — E3 and H2 depend on it; H2 with each WS; H3 last)
```

Start order: **A1, B1, H1 in the first pass** — manifest, registry format, and fixtures are the three things every other slice builds against. D1 (dev bundle refresh) is the first user-visible win and needs only A1+A3.

## 12. Gates — what "done" means

1. **The catch-up gate.** An H1 `0.9-pre-surfaces` fixture (edited variant) taken through `update` + a live `groundwork-upgrade` session ends with: current skills and dev bundle (byte-identical to fresh init), edited principles doc merged with intent preserved, surfaces registry bootstrapped, doc renames applied, every step a separate commit — and `groundwork check` reporting clean.
2. **The idempotency gate.** H2's invariants green in CI: convergence, preservation, idempotency, detect-honesty.
3. **The forcing-function gate.** G1 red on an unannotated change to the shipped surface; the next real release after this plan ships at least one registry-backed `[migration]` entry exercised by a fixture.
4. **The visibility gate.** A stale project's `groundwork check` names the version gap and pending work without network access, and `./dev doctor` flags a trailing bundle.

## 13. Execution notes (2026-06-12)

Deviations and refinements discovered while landing the plan; the tree is the truth,
this section explains where it differs from the sections above.

- **Tier-2 entries gained a `base` field** (refining O5): `base` is the hash of the
  *package* content a file was last reconciled against (null = unknown ancestry). A
  merge queues only when the package moves past `base` — without this, a user edit to
  a file the framework never changed queued a pointless merge. The upgrade skill
  re-bases the entry after each merge.
- **H1 fixtures are built from commit SHAs, not tags** — the repo has no release tags
  yet. `pre-0.9` ← `1498e0a` (genuinely unstamped: that CLI had no `stampVersion`);
  `0.9-pre-surfaces` ← `f81cc6b`. The dev bundle is not git-tracked (the unanchored
  `dist/` ignore), so the fixture's old bundle was *built* from the old commit's
  `cli-src`. Provenance in `tests/fixtures/installs/README.md`.
- **A fifth backfill brief shipped**: `gw-package-rename-invocations` (the
  `npx groundwork` → `npx groundwork-method` rename) — B3's cross-check flagged the
  existing changelog line the moment the convention became a test, which is the
  forcing function doing its job on day one. The B3 test scopes the id-suffix
  requirement to sections ≥ 0.10.0; older `[migration]` lines are prose history.
- **Two real cli migrations seed the registry**: `gw-seed-config-toml` and
  `gw-register-serena-mcp` — both heal things `init` does that `update` never did.
- **Migration completion is recorded for *settled* entries too**: a `detect` that
